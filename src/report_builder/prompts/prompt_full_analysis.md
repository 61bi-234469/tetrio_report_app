# TETR.IO 戦績分析レポート 生成プロンプト

> 使い方：以下の「===プロンプト本文===」以降をすべてコピーし、解析対象の
> TETR.IO Tetra League 戦績データ（原則 raw JSON。必要に応じて展開済みCSVも可）を添付して、
> **コード実行（Python）が使えるAI**に渡してください。
> コード実行がない環境では数値を捏造しやすいため、必ず実データを読み込み・集計させること。

---

## ===プロンプト本文===

あなたは対戦ゲーム **TETR.IO の戦績データアナリスト**です。添付された単一プレイヤーの Tetra League 戦績データを実際に解析し、**自己完結型のHTML分析レポートを1ファイル**作成してください。

推測ではなく、必ずPython（pandas、numpy、matplotlib等）で添付データを読み込み、集計結果から数値・図表・文章を生成してください。憶測値、未計算の概算、過去版からの無検証な数値流用は禁止です。

---

## 0. 最優先原則

1. **全数値を実データから算出し、主要値を中間出力して検算する**
2. **欠損している情報は捏造しない**。代替推定を行う場合は、式・前提・誤差・適用範囲を明記する
3. **勝敗そのものから作られる循環指標を勝因として扱わない**
4. **相関・予測と因果を区別する**。戦績だけから心理状態や最適戦略を断定しない
5. **小標本区分を一般化しない**。各表・グラフに標本数を併記する
6. 日付・曜日・時間帯・稼働日は、プレイヤーのローカル時間で集計する。日本在住なら **JST（UTC+9）** を用いる
7. 出力HTMLのファイル名は、生成日のローカル日付を `yyyy_mm_dd` 形式で必ず先頭に付ける
8. **図表を置くだけで終わらせない**。各グラフ・表について、読み方、実データ上の観察、解釈、注意点、実践上の意味を文章で説明する
9. 説明は「結論 → 数値的根拠 → 解釈 → 留保」の順を基本とし、読者がグラフを見なくても主旨を理解できる文章にする

## 1. 入力データとデータフレーム構築

### 1.1 raw JSON の想定構造

JSONは試合レコードの配列。主なフィールド：

- `ts`：試合日時（UTC、ISO 8601）
- `extras.result`：`victory` / `defeat` / `dqvictory` / `nullified`
- `extras.league[playerId]`：`[対戦前, 対戦後]` の2要素
  - `tr`, `glicko`, `rd`, `placement`, `rank` 等
- `results.leaderboard[]`：各プレイヤーの試合平均スタッツ
  - `apm`, `pps`, `vsscore`, `garbagesent`, `garbagereceived`, `kills`, `btb` 等
- `results.rounds[]`：ラウンドごと×プレイヤーごとの配列
  - `alive`, `lifetime`, `stats.apm`, `stats.pps`, `stats.vsscore` 等

対象プレイヤーは、全レコードの `extras.league` に共通して現れるIDを基本に特定し、相手はもう一方とする。自動特定が曖昧な場合は候補と根拠を表示する。

### 1.2 CSVの場合

展開済みCSVが添付された場合は、列名・粒度・単位を確認し、以下を明示する。

- 1行が試合単位かラウンド単位か
- 対象プレイヤー列と相手列の対応
- 日時・TR・勝敗・試合ID・ラウンド番号のマッピング
- JSON前提の項目で再現できないもの

### 1.3 必須DataFrame

最低限、以下の2つを構築する。

1. **試合レベルDataFrame**
   - 日時、結果、自分・相手のレーティング、TR増減、試合平均指標、期待勝率、セッション情報
2. **ラウンドレベルDataFrame**
   - 試合ID、ラウンド順、ラウンド勝敗、継続時間、自分・相手のラウンド指標、試合内点差

重複試合・重複ラウンド・時系列逆転・不正なゼロ除算・極端値を検査し、除外・補正件数をレポートに記録する。

## 2. 勝敗カウントと対象試合

- `dqvictory` は **勝利**として扱う。集計前に、対象プレイヤーのTR増減符号を実データで確認して裏づける
- `nullified` は勝率・モデル・連勝連敗から除外する
- 勝率は名称付きで併記する
  1. **公式戦績** = `(victory + dqvictory) / nullifiedを除く全試合`
  2. **通常完走試合のみ** = `victory / (victory + defeat)`
- 「不完全試合を除外した勝率」などの曖昧な表現は禁止
- 直近N戦勝率、期待超過勝数、連勝連敗は原則として公式戦績ベースで算出する
- DQ試合で試合スタッツが欠ける場合、勝敗集計には含め、能力指標分析からは欠損処理ルールを明示したうえで除外してよい

## 3. 指標定義

### 3.1 基本・派生指標

- **APM / PPS / VS**：入力値をそのまま使用
- **APP（Attack per Piece）**  
  `APP = APM / (PPS * 60)`
- **DS/S（Downstack per Second）**  
  `DS/S = (VS / 100) - (APM / 60)`
- **DS/P（Downstack per Piece）**  
  `DS/P = DS/S / PPS`
- **GbE（Garbage Efficiency）**  
  `GbE = ((APP * DS/S) / PPS) * 2`
- **Area**  
  `Area = APM + PPS*45 + VS*0.444 + APP*185 + DS/S*175 + DS/P*450 + GbE*315`
- **VS/APM**  
  `VS/APM = VS / APM`
  - APMが0または欠損なら欠損扱い
- **Est. TR**
  - 入力データに既存の推定TRがある場合のみ、その定義を確認して使用する
  - 独自推定する場合は、学習データ・式・評価誤差・適用範囲を明示し、実TRと混同しない
  - 根拠なく生成しない

### 3.2 4プレイスタイル

以下の定義を使用する。`srarea`、`statrank`、`dsp`、`vsapm` の入力元・変換方法を明記し、必要値がない場合は式を推測せず算出不能とする。

```text
nmapm = ((apm / srarea) / ((0.069 * pow(1.0017, (pow(statrank, 5) / 4700))) + statrank / 360)) - 1
nmpps = ((pps / srarea) / (0.0084264 * pow(2.14, (-2 * (statrank / 2.7 + 1.03))) - statrank / 5750 + 0.0067)) - 1
nmapp = (app / (0.1368803292 * pow(1.0024, (pow(statrank, 5) / 2800)) + statrank / 54)) - 1
nmdsp = (dsp / (0.02136327583 * pow(14, ((statrank - 14.75) / 3.9)) + statrank / 152 + 0.022)) - 1
nmgbe = (gbe / (statrank / 350 + 0.005948424455 * pow(3.8, ((statrank - 6.1) / 4)) + 0.006)) - 1
nmvsapm = (vsapm / (-pow(((statrank - 16) / 36), 2) + 2.133)) - 1

opener = ((nmapm + nmpps * 0.75 + nmvsapm * -10 + nmapp * 0.75 + nmdsp * -0.25) / 3.5) + 0.5
plonk = ((nmgbe + nmapp + nmdsp * 0.75 + nmpps * -1) / 2.73) + 0.5
stride = ((nmapm * -0.25 + nmpps + nmapp * -2 + nmdsp * -0.5) * 0.79) + 0.5
infds = ((nmdsp + nmapp * -0.75 + nmapm * 0.5 + nmvsapm * 1.5 + nmpps * 0.5) * 0.9) + 0.5
```

値を0〜1へ無断でクリップしない。可視化のために正規化する場合は、原値を別途示し、全系列で同一の正規化基準を使用する。

### 3.3 期待勝率

対戦前Glickoと相手RDがある場合、以下のGlicko近似を使用する。

```text
q = ln(10) / 400
g(RD) = 1 / sqrt(1 + 3*q^2*RD^2/pi^2)
E = 1 / (1 + 10^(-g(RD_opponent) * (r - r_opponent) / 400))
```

- TETR.IO内部のTR計算を完全再現するものではないと明記する
- 相手レーティング欠損試合は期待勝率分析から除外する
- Glicko/RDがなくTRだけがある場合、Glicko近似とは呼ばない
- TR差から期待勝率を較正する場合は、学習方法、時間分割、評価指標、較正誤差を明記し、**データ内較正モデル**として扱う

### 3.4 統計指標

- **効果量**：勝利群 vs 敗北群の Cohen's d
- **変動係数**：標準偏差 ÷ 平均
- **分位点**：P10 / P50（中央値）/ P90
- **期待超過勝率**：実績勝率 − 平均期待勝率
- **期待超過勝数**：実勝数 − 期待勝数合計
- 勝率の区間推定は、可能ならWilson区間またはブートストラップ区間を使用する

`kills`、最終スコア、取得ラウンド数など、結果とほぼ同義の変数は勝因モデルから除外する。

## 4. レポート目次と必須分析（8章）

各章は、単にグラフを並べるのではなく、次の順序で構成する。

1. **この章で見ること**：何を確認する章かを1〜2文で説明
2. **グラフまたは表**
3. **図の読み方**：軸、単位、色、系列、基準線、分母、標本数を説明
4. **観察された結果**：最大・最小、差、変化点、例外を実数値で示す
5. **解釈**：その結果がプレイ傾向や勝敗との関連について何を示唆するか
6. **注意点**：因果、逆因果、欠損、小標本、指標定義、選択バイアス等を示す
7. **要点（takeaway）**：読者が持ち帰るべき結論を1〜3文で明示

説明文は、原則として各グラフ・表につき**最低3〜5文**とする。複雑なグラフでは、読み方と結論を別段落に分ける。  
「上昇傾向が見られる」「差がある」のような抽象表現だけで済ませず、可能な限り数値・期間・比較対象を明記する。  
明確な差を確認できない場合は、無理に結論を作らず、**「このデータでは明確な差を確認できない」**と書く。

章を単純に追加するのではなく、以下の順序で重複なく整理する。

### 第1章　全体像と基本指標

- 対象期間、試合数、ラウンド数、稼働日数
- TR：開始、現在、ピーク、ピーク日、最大ドローダウン
- 公式戦績と通常完走試合の勝率
- 直近100 / 500 / 1000 / 全期間の実績勝率、期待勝率、期待超過勝数
- 初期配置直後のTR大変動は不確実性が大きいため、「純粋な実力上昇」と断定しない
- TR推移と主要イベントを1枚で俯瞰する
- TRグラフでは、開始値・現在値・ピーク値・最大下落幅を本文中でも再掲し、「長期傾向」と「短期変動」を分けて説明する
- 直近窓の勝率は、実績と期待の差が何勝分に相当するかまで文章化する

### 第2章　成長推移と安定性

平均値だけでなく、指標の推移とプレイ品質の再現性を分析する。

- 月次または一定試合数窓での基本・派生指標推移
- 正規化した複数指標の重ね描き
- 初期期間→直近期間の成長率
- TR、Area、4スタイル構成の変化
- 可能なら統計的変化点を補助的に検出するが、設定変更や練習内容との因果は断定しない
- 推移グラフでは、最も伸びた指標、停滞した指標、途中で方向が変わった指標を具体的に示す
- 正規化グラフでは、正規化後の線の高さではなく、元の単位での増減量も本文に併記する
- 期間別の **P10 / P50 / P90**
  - P10：不調時の下限水準
  - P50：通常時の水準
  - P90：好調時の上限水準
- 最初のN戦と直近N戦の変動係数を比較
- 「中央値の上昇」「下振れの底上げ」「上振れ天井の伸長」を分けて記述
- 変動を可能な範囲で分解（同一セッション内 / セッション間 / 日間 / 月間）
- 平均が伸びてもP10が伸びていない場合は、安定性向上とは書かない
- 箱ひげ図、分位帯付き時系列、変動係数比較を使用する
- P10・P50・P90の意味を図の直前または直後に平易に説明する
- 「平均が伸びた」のか、「不調時の底が上がった」のか、「好調時だけ伸びた」のかを必ず書き分ける
- 変動係数は値が小さいほど相対的に安定していることを説明し、平均値の変化と切り分けて解釈する

### 第3章　能力バランス

基本能力プロファイルと、プレイスタイル構成・相性を分析する。

- APM / PPS / VS / APP / DS/S / DS/P / GbE / Area / VS/APM の分布
- 自分と相手の平均・中央値・分位点比較
- 低PPSでも勝った試合の割合と、APP・GbE・Areaで補えているか
- 能力レーダーと4プレイスタイルレーダー
- 平均だけでなく分布の重なりを示し、「高い・低い」を単純化しすぎない
- レーダーチャートは正規化方法と基準集団を説明し、面積の大小をそのまま総合実力と解釈しない
- 各能力について「相手より高い／低い」だけでなく、その差をどの指標で補っているかを文章でつなぐ

#### 代表スタイル分類

- 原則として4スタイル値の最大項目を代表スタイルとする
- 1位と2位の差が小さい場合は「混合型」と注記し、分類の不確実性を示す
- 代表分類だけでなく、可能なら4スタイル連続値を使った補助分析も行う

#### 4×4相性表（補助）

- 行：自分の代表スタイル、列：相手の代表スタイル
- 順序は **Opener / Stride / Inf DS / Plonk** に統一する
- 各セルに試合数・実績勝率・平均期待勝率・期待超過勝率（必要に応じて信頼区間）を表示
- 主な評価値は生勝率ではなく、相手強度を補正した **期待超過勝率** とする
- 小標本セルはグレーアウトまたは注意表示し、順位づけしない
- 自分の各スタイル値と相手の各スタイル値の差が勝率へ与える関連
- 高PPS型、高APP型、高DS型など実測指標による相手タイプ分類との整合性
- 「PlonkはStrideに強い」などの一般論を先入観として当てはめず、このデータ内で確認できた範囲だけを書く
- 行が自分、列が相手であること、色が実績勝率か期待超過勝率かを明記する
- 生勝率が高くても期待勝率も高いセルは「相性が良い」と即断せず、期待超過を優先して解釈する

### 第4章　勝敗に関係しやすい指標

自分の絶対値だけでなく、対戦相手との差が勝敗にどう関連するかを分析する。

#### 必須の相対指標

- `ΔAPM = 自分APM − 相手APM`
- `ΔPPS`
- `ΔVS`
- `ΔAPP`
- `ΔDS/S`
- `ΔDS/P`
- `ΔGbE`
- `ΔArea`
- `ΔVS/APM`
- 算出可能なら `ΔOpener / ΔStride / ΔInf DS / ΔPlonk`

#### 必須分析

1. 各相対指標の勝利群・敗北群差とCohen's d
2. 指標差をビン分けした実績勝率と期待勝率
3. APM・VS優劣による4分類
   - APMのみ優位
   - VSのみ優位
   - 両方優位
   - 両方劣位
4. PPS劣位をAPP・DS/P・GbE優位で補える条件の分析
5. **ベースラインモデル**と**拡張モデル**の比較
   - ベースライン：対戦前レーティングまたは期待勝率のみ
   - 拡張：ベースライン＋相対能力指標
6. 可能なら時系列を保った訓練・検証分割を用い、AUC、Brier score、log loss、較正を比較
7. 多重共線性が強い指標は、相関行列・正則化・変数削減で扱う

モデルの係数や重要度は「勝敗を予測する関連性」であり、因果的な勝因とは断定しない。試合後に確定する情報や勝敗由来の変数を説明変数へ入れない。

モデル結果の説明では、次を本文中で明示する。

- どの相対指標が勝敗と強く関連したか
- 指標差がどの程度変わると、予測勝率がどの程度変わるか
- ベースラインから拡張モデルへ性能がどれだけ改善したか
- AUCは順位づけ能力、Brier score・log lossは確率予測の正確さを表すこと
- 性能改善が小さい場合は、「能力差を加えても追加説明力は限定的」と書く
- 係数の符号だけで重要度を判断せず、標準化、信頼区間、共線性を確認する

### 第5章　対戦相手の強さと期待値

- 相手TR差またはGlicko差別の実績勝率 vs 期待勝率
- 格上・同格・格下での期待超過勝率
- 直近100 / 500 / 1000 / 全期間の実績 vs 期待
- 格上で期待割れ、格下で期待超過などの非対称性
- 相手層の時期変化を示し、単純な月次勝率と混同しない
- 実績線と期待線の距離がどの帯で大きいかを数値で説明し、格上・同格・格下を一括りにしない

### 第6章　接戦・決着局面

接戦になった試合をどう閉じたかを、スコア状況・マッチポイント・タイブレークで分析する。

- スコア状況（ラウンド点差）別の次ラウンド勝率・最終勝率
- 先取マッチポイント後の勝率
- 逆転の定義と回数
  - 0-2開始からの逆転
  - 試合中2点差以上からの逆転

#### タイブレーク時の状況

ここでの**タイブレーク**は、単なる途中同点ではなく、**両者があと1ラウンド取れば試合勝利となる同点状態から始まる最終決着ラウンド**と定義する。
例：First to 3なら2-2、First to 4なら3-3。勝利条件が試合ごとに異なる場合は、各試合の目標ラウンド数から自動判定する。

最低限、次を分析する。

- タイブレーク到達試合数と全完走試合に占める割合
- タイブレーク勝率、平均期待勝率、期待超過勝率、期待超過勝数
- Wilson区間またはブートストラップ区間
- タイブレークへ入った経路別の成績
  1. 自分がリードしていたが追いつかれた
  2. 自分がビハインドから追いついた
  3. 大きな点差を経ず交互に取り合った
- 直前ラウンドの勝敗別
  - 直前ラウンドを取ってタイブレークへ入った場合
  - 直前ラウンドを落として追いつかれた場合
- 最終決着ラウンドのAPM / PPS / VS / APP / DS/P / GbE / Area / 継続時間を、通常ラウンドおよび同一試合のそれ以前のラウンドと比較
- タイブレークで普段より速度を上げて効率を落としていないか、または下掘り・生存寄りへ変化しているか
- タイブレーク勝利と、格上・同格・格下、代表プレイスタイル、セッション位置との交互作用
- 可能ならスコア推移を「追いついた側／追いつかれた側」で分けた勝率図または表を作る

タイブレークへ到達した試合だけを対象にするため、結果には**選択バイアス**がある。
標本数が少ない場合は「勝負強い／弱い」と断定せず、探索的結果として扱う。期待勝率を上回っていても区間が広い場合は、継続観察を優先する。

- 「引きずり」「勝負強さ」「クラッチ性能」等の表現は、期待勝率・標本数・相手強度を確認したうえで限定的に使う
- 心理状態そのものは戦績だけで断定しない
- 「敗戦後に弱い」等の表現を使う場合は、敗戦後の相手強度と期待勝率を調整した差も併記する
- 点差別勝率では、試合数が少ない大差区分より、標本数の多い区分を優先して説明する

### 第7章　ラウンド展開と試合時間

- 勝敗別のラウンド決着時間分布
- ラウンド継続時間別勝率を30秒粒度で示し、最低帯・最高帯を具体的に特定
- ラウンド決着時間帯別の能力差分（自分−相手）
- ラウンド時間と勝敗には逆因果があるため、「長いほど弱い」と単純解釈しない
- ラウンド時間別勝率では、最も低い帯・高い帯と標本数を示し、隣接帯との連続性も確認する

### 第8章　試合間・セッション全体の流れ

試合と試合の間で続く流れを、連勝連敗・前戦結果・セッション位置・ドローダウンで分析する。

- 連勝・連敗長の分布
- 勝利後・敗北後の次戦勝率（単発敗戦後と、2連敗・3連敗以降を分ける）
- セッション定義を明示する（例：前試合完了直後から次試合開始まで10分以内なら同一セッション）
- セッション内の試合位置別に、実績勝率・期待勝率・主要指標を比較
- ウォームアップ区間と終盤失速の有無
- セッション位置グラフでは、横軸が「そのセッションの何戦目か」であることを明記する
- TRドローダウンのアンダーウォーター図（0が過去最高TR、負値がピークからの下落幅）
- 最大ドローダウンの開始・底・回復日と、その間の能力指標変化を文章でまとめる
- 下落局面を次の2種類に分けて検討
  1. **コンディション低下型**：APM、APP、VS、Area等も低下
  2. **分散・結果下振れ型**：能力指標は維持し、期待比だけ下振れ
- 下落前後の指標差を示すが、原因を心理・疲労と断定しない
- 50戦程度の勝率は偶然だけでも約±7%動き得ることを踏まえ、「内容低下」と「結果の下振れ」を期待勝率・能力指標で切り分ける
- データに根拠がある範囲で、N戦ブロック、休憩、連敗ハードストップ等の運用案を示す

### 総合評価と改善優先順位（本文8章の後に置く）

- 強み・課題の星取りまたはスコアカード
- 改善優先順位表
  - 根拠指標
  - 推奨行動
  - 検証方法
  - 見直し条件
- 相手タイプ・連戦位置・劣勢状況別の実践的アドバイス
- 検証済みパーソナルレコードのうち重要な記録を本文で要約し、全表はレポート末尾の**付録B**へ折り畳みで掲載する（初期高不確実期の異常な単発TR増加は能力PRとして扱わない。PRは平均能力・単戦値・期間ベストを分けて表示する）

#### やらない方がよいこと

「分析資料を絶対視しない」といった読み方の注意だけで終わらせず、**実際のプレイングとセッション運用に関する内容を充実させる**。
次の2分類で整理し、合計5項目以上を目安とする。

##### A. データ・分析の扱いで避けること

例：

- 短期勝率や単一グラフだけで、実力低下・成長・心理状態を断定する
- レーダーチャートの面積やプレイスタイル分類を、絶対的な実力・正解戦術として扱う
- 小標本の相性セルやタイブレーク成績を一般化する
- 相関・モデル重要度を、因果的な勝因や「必ず伸ばすべき能力」と読み替える
- レポートの推奨値へ合わせるためだけに、設定やプレイスタイルを頻繁に変更する

##### B. 実際のプレイング・連戦運用で避けること

データで確認できた弱点に対応させ、次のような候補から**根拠のあるものだけ**を具体化する。

- PPSを上げるほどAPP・GbE・Area・勝率が悪化する速度帯で、無理に速度だけを上げ続ける
- APMを増やすことを優先し、VS、DS/P、盤面維持、相殺・下掘りを犠牲にする
- 高速型・高火力型など相手の特徴を無視し、毎試合同じ攻め方や同じプレイスタイルを押し通す
- 劣勢やタイブレークで、普段と異なる速度・攻撃偏重へ急激に切り替え、再現性を失う
- 単発敗戦後に取り返そうとしてテンポを上げすぎる、開幕や積みを急に変更する
- 3連敗以降やセッション終盤など、期待比・能力指標が明確に落ちる区間でも惰性で続ける
- 長時間ラウンドで勝率が落ちている場合に、火力勝負だけを継続し、掘り・相殺・盤面整理へ切り替えない
- 得意なAPP・GbE・Plonk等を、相手や盤面を無視して常時最大化しようとする
- 直近の好調・不調を根拠に、DAS/ARRや開幕構成を短期間で繰り返し変更する

各項目は、次の形式で書く。

```text
避けたい行動：
根拠となるデータ：
不利になり得る理由：
代わりに行うこと：
見直し条件：
```

実データで裏づけられないプレイング上の禁止事項は、断定的に書かない。
差が小さい、区間が広い、再現性がない場合は「避けるべき」ではなく「検証候補」として扱う。

- 提言はデータの範囲に限定し、「方向性は完全に正しい」「継続が最善」等の過度な断定を避ける
- 各提言は「観察された事実 → 解釈 → 推奨行動 → 検証方法」の順で書く
- 優先順位は、効果の大きさ、再現性、改善可能性、実行コストを分けて説明する

```html

  <summary>付録B　検証済みパーソナルレコードを開く</summary>

```

記録は元データから再計算し、日時・対象範囲・欠損を照合した**検証済みの値だけ**を掲載する。最低限、次の分類を設ける。

1. **レーティング記録**

3. **ラウンド記録**

4. **期間・窓記録**
   - 直近または連続N戦での最高勝率
   - N戦平均の最高APM / PPS / APP / Area等
   - 最長連勝・連敗
5. **試合展開記録**
   - 最大点差からの逆転
   - 0-2等からの逆転回数
   - タイブレーク勝利数・勝率

各記録に次の情報を付ける。

- 記録名
- 値と単位
- 発生日または対象期間（JST）
- 試合IDまたは追跡可能な識別子
- 対戦相手（データに存在し、表示上問題がない場合）
- 集計粒度（単試合／単ラウンド／N戦平均／月間）
- 検証条件と注意点
- 同率記録の件数

初期配置直後の極端なTR変動、欠損を0扱いして生じた値、ゼロ除算に近い派生指標、異常に短い無効ラウンドは能力PRから除外する。  
単発値とN戦平均を混同せず、再現性の高い期間記録を単発記録と分けて示す。

## 7. 図表の説明文仕様

### 7.1 各図表に必ず付ける説明

すべてのグラフ・表に、原則として次の5要素を付ける。

1. **図の読み方**
   - 横軸、縦軸、単位、色、線、基準線、正規化方法
   - 行と列の意味
   - 値が高い／低いと何を意味するか
2. **観察された事実**
   - 最大値、最小値、平均との差、前期比、相手との差
   - 変化点、ピーク、谷、例外
   - 標本数と欠損数
3. **解釈**
   - 何が強み・課題として示唆されるか
   - 他の指標とどうつながるか
   - 勝敗やTRとの関連がどの程度あるか
4. **注意点**
   - 因果ではなく関連であること
   - 逆因果、相手層、時期、自己選択、小標本、欠損の影響
5. **実践上の意味**
   - 次に何を維持・改善・検証すべきか
   - 現時点では行動提言に足る差か、それとも継続観察が必要か

説明は定型文の貼り付けではなく、**その図で実際に得られた数値に合わせて書き分ける**こと。

### 7.2 推奨する文章構造

各図表の直後に、次の順で文章を置く。

```text
【読み方】
この図は何を比較しているか。軸・色・単位・標本数は何か。

【結果】
最も重要な数値差や変化は何か。具体的な値を最低2つ示す。

【解釈】
その結果は何を示唆するか。他の指標との整合性はあるか。

【注意】
因果、逆因果、小標本、欠損、正規化等の制約は何か。

【要点】
読者が持ち帰るべき結論を1〜2文で示す。
```

HTML上では、見出しラベルをそのまま表示してもよい。  
文章量が過剰にならないよう、単純な図では3〜5文、複雑な図では5〜9文を目安とする。

### 7.3 結論の書き方

結論は次の順序を基本とする。

```text
結論 → 数値的根拠 → 解釈 → 留保
```

良い例：

> 直近300戦では、通常時のAPPだけでなく不調時の下限も改善している。P50は0.56から0.61、P10は0.45から0.51へ上昇した。一方、P90の伸びは小さく、上振れの拡大より再現性の改善が中心とみられる。ただし、直近期間は相手TRが平均120高いため、単純な期間比較には相手層の違いが含まれる。

避ける例：

> APPは上昇傾向です。良い成長だと考えられます。

### 7.4 結論の信頼性の扱い

確度ラベル（確度：高／中／低）は付けない。結論の確からしさは、標本数、区間推定、効果量、再現性を本文中で具体的な数値として示すことで表現する。

### 7.5 グラフ種類ごとの説明要件

#### TR推移・ドローダウン

- 開始、現在、ピーク、最大下落を本文で数値化
- 上昇期、停滞期、回復期を期間で区切る
- 初期配置の大変動と、その後の安定した変化を分ける
- ドローダウン図では0線と負値の意味を説明する

#### 分布・箱ひげ・分位帯

- 平均だけでなく中央値、P10、P90を説明
- 分布の重なりが大きい場合は「差はあるが判別力は限定的」と書く
- 外れ値が全体傾向を歪めていないか確認する

#### レーダーチャート

- 正規化基準と基準集団を説明
- 形状は相対的な偏りであり、面積は総合実力そのものではないと明記
- 最も突出した軸、低い軸、それらの組み合わせを文章化する
- 自分と相手を重ねる場合は、差が大きい軸を実数値でも示す

#### ヒートマップ

- 行、列、色、数値、グレーセルの意味を明記
- 最良・最悪セルだけでなく、標本数を併記
- 生勝率と期待超過を混同しない
- 小標本セルは結論の中心にしない

#### 実績勝率 vs 期待勝率

- 2本の線または棒が何を表すか説明
- 差をパーセントポイントと期待超過勝数で示す
- 期待線より高い／低いだけでなく、区間推定と標本数を確認する

#### 勝敗モデル・重要度

- 予測対象、説明変数、訓練期間、検証期間を説明
- AUC、Brier score、log lossの意味を平易に説明
- ベースラインとの差を示し、実用上の改善幅を評価する
- 重要度は因果的勝因ではないと繰り返し明記する

#### 移動平均・ノイズ帯

- 窓幅と帯の計算方法を明記
- 短期の上下と長期の傾きを分ける
- 帯を外れた区間でも、相手層や能力指標を確認してから結論づける

#### 試合展開・タイブレーク

- 点差の符号が自分視点か相手視点かを明記する
- タイブレークの勝利条件と抽出条件を説明する
- 追いついた側／追いつかれた側、直前ラウンド勝敗、期待勝率を分けて示す
- タイブレークだけを抽出する選択バイアスと小標本に注意する
- 「勝負強さ」「クラッチ性能」は、期待超過と区間推定が明瞭な場合に限って使う

#### セッション位置・ラウンド時間

- 各ビンの定義と試合数を示す
- 隣接ビンと連続した傾向か、単独の偶然値かを確認する
- ラウンド時間には逆因果があることを本文内でも説明する

### 7.6 可視化内の補助

読み取りやすさを高めるため、必要に応じて次を用いる。

- ピーク、最低点、変化点への注釈
- 実績と期待の差を示すラベル
- 標本数 `n=` の表示
- 基準線、平均線、0線
- 小標本区分の薄色・網掛け
- 図タイトルの直下に1文のサブタイトル

ただし、注釈を増やしすぎて図を読みにくくしない。詳細は本文で補う。

## 8. 分析の厳密性と注意書き

### 8.1 循環・データリーク

- `kills` は取得ラウンド数とほぼ同義になり得るため、「撃墜数が最大の勝因」と書かない
- 最終スコア、勝者フラグ、試合終了後の情報を勝敗予測モデルへ入れない
- 同一試合のラウンドが訓練・検証へ分散しないよう、モデル分割は試合単位で行う
- 将来予測の評価では、可能な限り時系列順を保つ

### 8.2 因果の注意

必ず以下をフッターまたは各章に注記する。

- ラウンド時間と勝敗には逆因果がある。苦しい試合ほど長引く可能性がある
- 時間帯、曜日、連戦位置には、プレー時期・相手層・自己選択の偏りがある
- 試合スタッツはセット平均であり、個別の開幕、盤面、相殺、入力ミス、回線状態は識別できない
- プレイスタイル分類は数式上の特徴量であり、実際の意思決定や開幕選択を直接観測したものではない
- 勝敗モデルは予測上の関連を示すもので、因果効果を証明しない
- タイブレーク分析は「接戦を最終局面まで生き残った試合」に条件づけられており、通常試合との単純比較には選択バイアスがある

### 8.3 標本数と多重比較

- 各区分に試合数を併記する
- 少数セルは非表示・グレーアウト・参考値扱いにする
- 多数の指標・区分を探索した場合、偶然の有意差が含まれることを明記する
- p値だけでなく、効果量、区間推定、再現性を優先する

### 8.4 ノイズと実力

- 約50戦の勝率は偶然だけでも±約7%動き得る
- 日々の勝敗より、長期TRの傾き、期待超過勝数、月次指標、P10/P50/P90、相対指標を重視する
- 単発敗戦と連敗累積を区別する

## 11. 推奨実行手順

1. 入力ファイルを読み、データ構造・列・粒度を確認する
2. 対象プレイヤーを特定する
3. 試合レベルとラウンドレベルのDataFrameを構築する
4. 重複、欠損、外れ値、時系列、DQのTR符号を検証する
5. 勝敗ルールを確定する
6. 基本・派生指標、4スタイル、期待勝率を計算する
7. 主要件数・平均・勝率を中間出力して自己検証する
8. 8章の順序に従って集計する
9. 試合ごとの勝利条件を確認し、タイブレークを「双方あと1ラウンドで勝利」の決着局面として抽出・検証する
10. 相対勝敗モデルはデータリークを避け、時系列分割で評価する
11. グラフの固定軸順・スタイル配置を確認する
12. 各グラフについて、読み方・結果・解釈・注意・要点の文章を実データから生成する
13. 各説明文に、最低2つの具体的数値または比較結果が含まれているか確認する
14. 全グラフをbase64化し、HTMLへ埋め込む
15. 月別集計と検証済みパーソナルレコードを再検算し、折り畳み式付録として組み込む
16. KPI、目次、8章、付録、テーブル、説明文、注意書きを1つのHTMLへ組み立てる
17. `yyyy_mm_dd_`で始まるファイル名で保存する
18. HTMLを実際に開き、画像欠落、文字化け、表崩れ、折り畳み動作、数値不整合を最終確認する
19. グラフを非表示にして文章だけ読んでも、主要結論が理解できるか最終確認する

## 13. 最終品質チェック

完成前に、以下を自己点検する。

- 各グラフ・表に「何を見る図か」の説明がある
- 軸、単位、色、系列、標本数が本文またはキャプションで分かる
- 各グラフの説明に具体的な数値が含まれる
- 最も重要な差、最小値、最大値、変化点が文章化されている
- 「差がある」と「明確な差を確認できない」を適切に使い分けている
- レーダー、ヒートマップ、期待勝率、モデル評価の読み方を説明している
- 結論と注意点が分離されている
- 小標本や推定依存の結果へ強い表現を使っていない
- 各章末のtakeawayが、その章の図表と一致している
- タイブレークを単なる途中同点と混同せず、双方があと1ラウンドで勝利する決着局面として抽出している
- タイブレーク成績に試合数、期待勝率、区間推定、到達経路の説明がある
- 「やらない方がよいこと」に、分析資料の読み方だけでなく、データに根拠を持つプレイング・連戦運用上の項目が含まれる
- 各「やらない方がよいこと」に、根拠・理由・代替行動・見直し条件がある
- 付録Aと付録Bがレポート末尾にあり、初期状態で閉じた折り畳みとして正常に動作する
- 月別集計の勝率・TR・主要指標が本文の集計値と一致する
- パーソナルレコードが元データから検証され、異常値・欠損・初期不確実期の扱いが明記されている
- グラフを見なくても、文章だけで主要結論を理解できる
